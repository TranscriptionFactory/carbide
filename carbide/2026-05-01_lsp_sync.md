Finding 1: Completion race condition (didChange vs completion timing)

 Problem: textDocument/didChange is debounced at 300ms
 (lsp_document_sync.reactor.svelte.ts). Completion requests debounce at 200ms
 (lsp_completion_plugin.ts). When a user types a trigger character, the completion
 request can fire ~100ms before didChange reaches the LSP server, causing the server to
 return suggestions based on stale document content.

 Recommendation: Flush the pending didChange before sending completion. Two options:
 - Option A (simple): Increase completion debounce to 350ms+ (ensures didChange fires
 first). Slightly slower UX.
 - Option B (correct): Before calling on_completion(), trigger an immediate didChange
 flush. This requires the completion plugin to have access to a flush_did_change()
 callback from the document sync reactor.

 I'd recommend Option B for correctness but this is a separate PR scope.
