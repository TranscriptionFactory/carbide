## Existing Architecture

**Backend (Rust):** `src-tauri/src/features/markdown_lsp/` spawns `markdown-oxide` as a subprocess using the generic `RestartableLspClient` from `src-tauri/src/shared/lsp_client/`. Communication is standard LSP over stdio (JSON-RPC). One client instance per vault, keyed in a `HashMap<String, RestartableLspClient>`.

**Frontend (TypeScript):** The ports/adapters pattern:

- **Port:** `src/lib/features/markdown_lsp/ports.ts` — defines the interface (hover, completions, references, rename, etc.)
- **Adapter:** `src/lib/features/markdown_lsp/adapters/markdown_lsp_tauri_adapter.ts` — calls `tauri_invoke()` for each method
- **Service:** `src/lib/features/markdown_lsp/application/markdown_lsp_service.ts` — orchestrates lifecycle, document sync, diagnostics
- **Store:** `src/lib/features/markdown_lsp/state/markdown_lsp_store.svelte.ts` — reactive `$state`
- **Reactor:** `src/lib/reactors/markdown_lsp_lifecycle.reactor.svelte.ts` — watches vault/settings, starts/stops LSP automatically

**Editor integration:** ProseMirror plugins in `src/lib/features/editor/adapters/lsp*.ts` handle hover tooltips, completions, go-to-definition, code actions, and inlay hints.

## Data Flow (e.g., Hover)

```
User hovers → ProseMirror plugin extracts (line, char)
  → tauri_invoke("markdown_lsp_hover", {vault_id, file_path, line, char})
  → Rust finds RestartableLspClient → sends textDocument/hover to markdown-oxide
  → Response flows back → plugin renders tooltip via @floating-ui
```

## Current State: Honest Assessment

**The generic layer is solid** — `shared/lsp_client/` (transport, restartable client, types) is well-factored and reusable. The frontend editor plugins (`lsp_hover_plugin`, `lsp_completion_plugin`, etc.) are also provider-agnostic.

**But adding a new LSP still has significant friction.** Here's what's duplicated or tightly coupled:

### Rust Backend — Duplicated Code

| Concern                 | Locations                                                       | Issue                                                                   |
| ----------------------- | --------------------------------------------------------------- | ----------------------------------------------------------------------- |
| `percent_decode()`      | `markdown_lsp/service.rs`, `code_lsp/manager.rs`, `lint/lsp.rs` | **3 copies** of the same function                                       |
| Diagnostic parsing      | `parse_lsp_diagnostics()` vs `parse_diagnostic()`               | Same LSP JSON → struct conversion, different output types               |
| URI → path conversion   | `uri_to_path()` vs `uri_to_relative_path()`                     | Both do file URI parsing with slight variations                         |
| Notification forwarding | `spawn_notification_forwarder()` vs `forward_notifications()`   | Nearly identical — filter `publishDiagnostics`, parse, emit Tauri event |
| Status forwarding       | `spawn_status_forwarder()` vs `forward_status()`                | Same pattern                                                            |

### Rust Backend — markdown_lsp/service.rs is a monolith

That file has ~1600 lines with **50+ top-level functions**. It contains:

- Generic LSP plumbing (document sync, position helpers, workspace edit application)
- Markdown-specific provider resolution
- IWE-specific config management (~300 lines of TOML rewriting)
- All `#[tauri::command]` wrappers

Adding Markdown Oxide here means this file grows further, and adding Harper later means duplicating the plumbing a third time.

### Frontend — Less Problematic

The frontend is cleaner. The ports/adapters/service/store pattern means each LSP feature is isolated, and the editor plugins are already provider-agnostic. The shared `lsp/` feature has common types and the workspace edit applier. This side would need minimal refactoring.

### What a "Low Friction" Architecture Looks Like

Before adding Markdown Oxide, I'd recommend extracting the shared Rust plumbing:

1. **`shared/lsp_client/utils.rs`** — Move `percent_decode`, `uri_to_path`, `file_uri()`, severity mapping there. Delete all 3 copies.

2. **`shared/lsp_client/diagnostics.rs`** — Generic `parse_publishDiagnostics(params) → Vec<LspDiagnostic>` with a shared diagnostic type. Feature modules map to their own types if needed.

3. **`shared/lsp_client/forwarding.rs`** — Generic notification/status forwarder that takes an event name and a parser closure. Eliminates copy-paste for each new LSP.

4. **Split `markdown_lsp/service.rs`** — Extract IWE config management into its own module, and pull generic LSP command wrappers (did_open, did_change, hover, etc.) into reusable helpers parameterized by state key.

After that extraction, adding a new LSP (Markdown Oxide, Harper, etc.) becomes:

- Define the enum variant + capabilities
- Add a provider resolution arm
- Register the Tauri commands (which are thin wrappers over the shared helpers)
- Wire up frontend port/adapter/store (already low friction)

**Bottom line**: The generic transport layer is good, but the feature layer above it has organic duplication that makes each new LSP more work than it should be. A focused refactor of the Rust side before adding Markdown Oxide would pay for itself immediately with the second (and third) addition.
