## Claude

Three are worth considering, in order of value:

---

### 1. Streamable HTTP (high value, low effort)

The [2025-03-26 MCP spec update](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports) replaced the old HTTP+SSE transport with **Streamable HTTP** — a single `/mcp` endpoint that handles both request/response and optional SSE streaming in the same connection. Claude Code already supports it.

Carbide's current HTTP transport is a stateless subset of this: POST → JSON response. Adding Streamable HTTP means:
- For short calls: same as now (200 + JSON body)
- For long-running tools (index rebuild, semantic search over many notes): the server can stream SSE progress events before the final result

Concretely, that means adding `Accept: text/event-stream` detection in the `mcp_handler`, a session ID header, and an optional SSE upgrade path. The Axum infrastructure is already there. This is probably 1 unit of work and the highest-leverage change — it future-proofs the transport and enables progress streaming for slow tools.

---

### 2. stdio via CLI proxy (medium value, medium effort)

The current `mcp_start` / stdio server can't work in the Tauri process (GUI owns stdout). But `carbide-cli` already exists and communicates with the HTTP server. You could add a `carbide mcp` subcommand that:

1. Calls `ensure_running()` (already implemented — launches the app if not running)
2. Then proxies stdin/stdout JSON-RPC to `http://localhost:3457/mcp`

Claude Desktop config would become:
```json
{ "command": "/usr/local/bin/carbide", "args": ["mcp"] }
```

This is the correct architecture for Claude Desktop's traditional stdio flow (it spawns a child process with pipes). The proxy itself is ~50 lines. The main cost is: it requires `carbide` to be installed on PATH, which means the setup button should also trigger `--install-cli`.

---

### 3. WebSocket (low value)

Not in the MCP spec. A few third-party clients use it, but the spec direction is clearly Streamable HTTP. Not worth building.

---

**Recommendation:** Implement Streamable HTTP first (it unblocks streaming + aligns with spec), then add `carbide mcp` CLI proxy to fix the Claude Desktop stdio story properly. The current HTTP-only setup is functional for Claude Code; Desktop works if it's on a recent enough version that supports HTTP MCP.

Start broad, narrow progressively:

```

mcp__serena__find_symbol(name_path_pattern="ClassName/method_name")

# Substring match (don't know exact name)
mcp__serena__find_symbol(name_path_pattern="process", substring_matching=True, relative_path="src/")

mcp__serena__find_symbol(name_path_pattern="ClassName", depth=1)

```

mcp__serena__find_symbol(
    name_path_pattern="ClassName/method_name",
    relative_path="path/to/file.py",
    include_body=True
)

# Read with docstring/signature info (lighter than body)
mcp__serena__find_symbol(
    name_path_pattern="ClassName/method_name",
    include_info=True
)
```

### 5. Trace Dependencies

```

mcp__serena__find_referencing_symbols(
    name_path="function_name",
    relative_path="path/to/file.py"
)
```

After a non-trivial search sequence, ALWAYS call:
```
mcp__serena__think_about_collected_information()
```

This is mandatory — it ensures you've gathered enough before proceeding.

## Common Analysis Patterns

### Understand a Module
1. `get_symbols_overview` on the file (depth=1 for classes)
2. Identify key classes/functions from the overview
3. `find_symbol` with `include_info=True` on the important ones
4. `find_referencing_symbols` on exports to see who depends on this module
5. `think_about_collected_information`

### Trace a Call Chain
1. `find_symbol` to locate the entry point
2. `find_symbol(include_body=True)` to read what it calls
3. Follow each call: `find_symbol` on the called functions
4. Repeat until you reach the leaf functions
5. `think_about_collected_information`

### Assess Impact of a Change
1. `find_symbol` on the symbol you plan to change
2. `find_referencing_symbols` to find all callers/users
3. For each reference, check if the change is breaking
4. `think_about_collected_information`
5. Report: what breaks, what's safe, what needs updating

### Plan a Refactor
1. Use `mcp__sequential-thinking__sequentialthinking` to plan the approach
2. Explore relevant symbols with the workflow above
3. Map the dependency graph (what depends on what)
4. Identify the order of changes (leaves first, or top-down)
5. Write the plan to memory for the editing phase:
   ```
   mcp__serena__write_memory(
       memory_file_name="refactor-plan-<feature>",
       content="## Refactor Plan\n..."
   )
   ```

### Investigate a Bug
1. Start from the symptom — `search_for_pattern` on error messages or suspicious values
2. `find_symbol` on the function where the bug manifests
3. `find_symbol(include_body=True)` to read the logic
4. `find_referencing_symbols` to check callers for bad inputs
5. `think_about_collected_information`

     7→    reader: R,
     8→    mut writer: W,
     9→    router: std::sync::Arc<tokio::sync::Mutex<McpRouter>>,
    10→    shutdown: tokio::sync::watch::Receiver<bool>,

    18→

    20→        tokio::select! {
    21→            line = lines.next_line() => {
    22→                let line = line.map_err(|e| format!("IO error: {}", e))?;
    23→                let Some(line) = line else {
    24→                    log::info!("MCP transport: EOF on input stream");
    25→                    break;
    26→                };

    30→                    continue;
    31→                }
    32→

    34→                    Ok(request) => {
    35→                        let mut r = router.lock().await;
    36→                        r.handle_request(&request)
    37→                    }
    38→                    Err(err_response) => Some(err_response),

    43→                        .map_err(|e| format!("Serialization error: {}", e))?;
    44→                    writer.write_all(serialized.as_bytes()).await

### Part 1: Streamable HTTP Transport

**Branch:** `feat/mcp-streamable-http`  
**Depends on:** current `feat/extended-tools` (HTTP server is running)

---

#### What Streamable HTTP actually requires

The [2025-03-26 MCP spec](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports#streamable-http) mandates:

- **POST `/mcp`** — client sends JSON-RPC, server may respond with either:
  - `Content-Type: application/json` — single response (current behavior)
  - `Content-Type: text/event-stream` — SSE stream (when `Accept: text/event-stream` sent)
- **GET `/mcp`** — client opens a persistent SSE channel for server-initiated notifications
- **`Mcp-Session-Id`** header — ties multiple requests together; server issues it on `initialize`, client echoes it

Phase 1 (one session): handle POST streaming + session ID. Skip GET (server push) — Carbide has no server-initiated events yet.

   pub async fn run_stdio_proxy(client: &CarbideClient) -> Result<(), String> {
       use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
       let stdin = tokio::io::stdin();
       let mut stdout = tokio::io::stdout();
       let mut lines = BufReader::new(stdin).lines();
   
       while let Some(line) = lines.next_line().await.map_err(|e| e.to_string())? {
           let trimmed = line.trim();
           if trimmed.is_empty() { continue; }
   
           let response = client.post_mcp(trimmed).await?;
           stdout.write_all(response.as_bytes()).await.map_err(|e| e.to_string())?;
           stdout.write_all(b"\n").await.map_err(|e| e.to_string())?;
           stdout.flush().await.map_err(|e| e.to_string())?;
       }
       Ok(())
   }

   pub async fn post_mcp(&self, body: &str) -> Result<String, String> {
       let resp = self.http
           .post(format!("{}/mcp", BASE_URL))
           .bearer_auth(&self.token)
           .header("Content-Type", "application/json")
           .body(body.to_owned())
           .send().await.map_err(|e| e.to_string())?;
   
       let content_type = resp.headers()
           .get("content-type")
           .and_then(|v| v.to_str().ok())
           .unwrap_or("");
   
       let text = resp.text().await.map_err(|e| e.to_string())?;
   
       if content_type.contains("text/event-stream") {
           // Extract JSON from "data: {...}" line
           Ok(text.lines()
               .find(|l| l.starts_with("data: "))
               .map(|l| l[6..].to_owned())
               .unwrap_or(text))
       } else {
           Ok(text)
       }
   }
   ```

### Part 3: Terminal — Bugs and Suboptimal Details

Three real bugs found, one design issue:

#### Bug 1: Session tab switch destroys xterm state

**File:** `terminal_panel_content.svelte:155–159`

```svelte
{#each session_ids as session_id (session_id)}
  {#if session_id === active_session_id}   <!-- WRONG: destroys inactive sessions -->
    <TerminalSessionView {session_id} active={true} />
  {/if}
{/each}
```

When switching tabs, the inactive session's `TerminalSessionView` is removed from the DOM — `onDestroy` fires, `terminal.dispose()` is called, and xterm's scrollback buffer is gone. Switching back creates a blank terminal.

**Fix:** always render all session views; use `active` prop (which already controls CSS `display`) to show/hide:
```svelte
{#each session_ids as session_id (session_id)}
  <TerminalSessionView {session_id} active={session_id === active_session_id} />
{/each}
```
The `TerminalSessionView` CSS already handles this: `.TerminalSessionView { display: none }` and `.TerminalSessionView--active { display: block }`. The `active` prop gate in the component already prevents input from leaking to background sessions. The xterm instance persists, scrollback is preserved.

#### Bug 2: `fixed_cwd` ignores stored session cwd

**File:** `terminal_session_view.svelte:39–44`

#### Bug 3: Toggle hides panel but kills all PTY processes

**File:** `terminal_actions.ts:47`, `62`

```ts
terminal_service.close_all_sessions();  // sends SIGKILL to all shells
ui_store.bottom_panel_open = false;
```

Toggling the terminal panel (or clicking X) kills every running shell process. Re-opening creates blank terminals. Sessions are meant to persist across panel show/hide — otherwise multi-session tabs are useless.

**Fix:** separate "hide panel" from "kill sessions". Introduce `hide()` and `show()` on `TerminalStore` (set `panel_open` without touching `sessions`):

```ts
// terminal_toggle action:
if (ui_store.bottom_panel_open && ui_store.bottom_panel_tab === "terminal") {
  ui_store.bottom_panel_open = false;           // just hide
  return;
}
ui_store.bottom_panel_tab = "terminal";
ui_store.bottom_panel_open = true;
terminal_store.open();                          // ensure panel state consistent

// terminal_close action:
ui_store.bottom_panel_open = false;             // also just hide
// only kill if user explicitly closes all session tabs
```

The `close_all_sessions` path should only be hit when the user individually closes every tab (which already fires `terminal_close_session` per session). Introduce a `terminal_kill_all` action if the user explicitly wants to nuke everything (can be a keyboard shortcut or context menu item).

`TerminalStore.close()` also needs splitting: `close()` → just `panel_open = false; focused = false` without clearing `sessions`. Add `reset()` as the destructive version (keep current `close` body).

#### Design issue: `reconcile_session` always respawns for any setting drift

**File:** `terminal_reconcile.reactor.svelte.ts:88`

`reconcile_session` is called whenever the reactor detects `!already_aligned`. It calls `respawn_session`, which unconditionally kills and restarts. For sessions with `respawn_policy: "manual"`, this is wrong — only `respawn_policy: "on_context_change"` sessions should auto-respawn.

**Fix in `reconcile_session`:**
```ts
async reconcile_session(session_id, input): Promise<string> {
  const session = this.terminal_store.get_session(session_id);
  if (session?.respawn_policy === "manual") {
    // Only update stored metadata, don't kill the process
    this.terminal_store.ensure_session({ id: session_id, ...input }, { activate: false });
    return session_id;
  }
  return this.respawn_session(session_id, { ... });
}
```
